# QuizCraft

**QuizCraft** — веб-платформа для создания, редактирования и прохождения квизов/викторин. Проект ориентирован на преподавателей, тренеров и студентов: преподаватель создаёт банк вопросов и квизы, участники проходят попытки, а система автоматически считает результаты и собирает аналитику.

Проект выполнен как программная система: есть frontend на React, backend на Django REST Framework, PostgreSQL в Docker и готовый сценарий демонстрационного наполнения.

## Демо-данные

В проекте есть management command для автоматического создания демонстрационных данных:

```bash
docker compose exec backend python manage.py seed_demo
```

Команда создаёт:

- пользователей;
- темы вопросов;
- банк вопросов;
- CSV-файл для импорта;
- демонстрационные квизы;
- приватный квиз;
- квиз со скрытым результатом;
- live-квиз как задел под расширение;
- черновик;
- архивный квиз.

Если нужна полностью чистая база перед демонстрацией:

```bash
docker compose down -v
docker compose up -d --build
docker compose exec backend python manage.py seed_demo
```

Флаг `-v` удаляет Docker volume PostgreSQL, то есть вся старая база будет очищена.

## Основные сценарии использования

### 1. Регистрация и вход

Пользователь может зарегистрироваться, указав:

- имя;
- логин;
- email;
- пароль.

После регистрации пользователь может входить по логину или email.

### 2. Создание банка вопросов

Преподаватель переходит в раздел «Банк вопросов» и создаёт вопросы разных типов. Для вопроса можно задать:

- текст;
- тип;
- баллы;
- тему;
- теги;
- цель обучения;
- объяснение после ответа;
- правильный ответ;
- варианты ответа;
- медиа-файл или ссылку на медиа.

### 3. Создание квиза

Преподаватель создаёт квиз и настраивает:

- название;
- описание;
- публичность или приватность;
- список пользователей, которым доступен приватный квиз;
- сложность;
- статус;
- время на прохождение;
- количество попыток;
- политику обратной связи;
- режим проведения;
- перемешивание вопросов;
- перемешивание вариантов;
- файлы квиза.

В квиз можно добавить вопросы из банка или создать новые вопросы вручную.

### 4. Прохождение квиза

Участник открывает доступный квиз из каталога или переходит по коду доступа.

При старте создаётся попытка. Система фиксирует:

- пользователя;
- квиз;
- время начала;
- порядок вопросов;
- порядок вариантов;
- дедлайн, если задан таймер.

После отправки система автоматически проверяет ответы и сохраняет результат.

### 5. Просмотр результата

В зависимости от политики обратной связи участник видит:

- полный разбор;
- только итоговый балл;
- только факт завершения без результата.

### 6. Аналитика

Преподаватель открывает аналитику квиза и видит:

- количество попыток;
- средний результат;
- минимальный результат;
- максимальный результат;
- список попыток;
- точность по вопросам;
- самые сложные вопросы;
- экспорт результатов в CSV.

## Типы вопросов

Поддерживаются следующие типы вопросов.

### Один правильный вариант

Пользователь выбирает один вариант из списка. Баллы начисляются, если выбран правильный вариант.

### Несколько правильных вариантов

Пользователь выбирает несколько вариантов. Поддерживается частичный зачёт: если выбрана часть правильных вариантов, начисляется часть баллов. Лишние неправильные варианты снижают результат.

### Верно/неверно

Вопрос с двумя вариантами ответа:

```text
Верно
Неверно
```

### Краткий ответ

Пользователь вводит текстовый ответ. Проверка выполняется по правильному текстовому значению.

### Числовой ответ

Пользователь вводит число. Для проверки можно задать:

- правильное число;
- допустимое отклонение.

Например:

```text
Правильное число: 3.14
Допуск: 0.01
```

Тогда ответы `3.13`, `3.14`, `3.15` могут считаться допустимыми.

Основные поля:

| Поле | Описание |

| `tags` | теги |

| `explanation` | объяснение после ответа |

| `media_url` | ссылка на медиа |
| `options` | варианты ответа через `|` |
| `correct_options` | номера правильных вариантов через `|` |

## Аналитика

В приложении есть два уровня аналитики.

### 1. Профильная статистика пользователя

В профиле показываются:

- количество созданных квизов;
- количество вопросов в банке;
- количество всех прохождений;
- количество завершённых прохождений;
- средний результат;
- среднее количество квизов в месяц за последний год;
- среднее количество вопросов в месяц за последний год;
- среднее количество прохождений в месяц за последний год;
- графики активности за 14 дней;
- месячная и годовая сводка.

### 2. Аналитика конкретного квиза

На странице аналитики квиза показываются:

- количество попыток;
- средний результат;
- минимальный результат;
- максимальный результат;
- список попыток участников;
- точность по каждому вопросу;
- список сложных вопросов;
- экспорт результатов в CSV.

## Сброс пароля

На странице входа есть ссылка:

```text
Забыли пароль? Сбросить пароль
```

Сценарий:

1. Пользователь вводит email или логин.
2. Backend генерирует одноразовый токен.
3. В режиме разработки ссылка показывается в dev-ответе или выводится в логи backend.
4. Пользователь переходит по ссылке.
5. Вводит новый пароль два раза.
6. После успешной смены пароля входит уже с новым паролем.

Посмотреть логи backend:

```bash
docker compose logs backend -f
```

В production-режиме вместо вывода ссылки в консоль можно подключить SMTP-сервер и отправлять письмо на email пользователя.

## Локальная разработка без Docker

Основной рекомендуемый способ запуска — Docker Compose. Но backend можно запустить локально.

### Backend локально

Создать виртуальное окружение:

```bash
python -m venv venv
source venv/bin/activate
```

Установить зависимости:

```bash
pip install -r requirements.txt
```

Применить миграции:

```bash
python manage.py migrate
```

Запустить backend:

```bash
python manage.py runserver
```

Без переменной `DATABASE_URL` проект использует SQLite:

```text
db.sqlite3
```

### Frontend локально

## Частые проблемы

### 1. Docker не может скачать образ

Ошибка может выглядеть так:

```text
TLS handshake timeout
```

Обычно это проблема доступа к Docker Hub. Можно попробовать:

```bash
docker pull python:3.11-slim
docker pull node:20
```

Также помогают:

```bash
docker logout
docker login
```

или перезапуск Docker Desktop.

### 2. Контейнеры не остановились после `docker compose down`

`docker compose down` останавливает контейнеры текущего compose-проекта. Если проект запускался из другой папки, контейнеры могут остаться.

Можно удалить вручную:

```bash
docker rm -f quizcraft_frontend quizcraft_backend quizcraft_db
```

### 3. Нужно полностью очистить базу

```bash
docker compose down -v
docker compose up -d --build
```

После этого можно заново создать демо-данные:

```bash
docker compose exec backend python manage.py seed_demo
```

### 4. Аватар или media-файл не отображается

Проверить, существует ли файл:

```bash
docker compose exec backend ls -R /app/media
```

Проверить прямую ссылку:

```text
http://localhost:8000/media/...
```

и ссылку через frontend:

```text
http://localhost:5173/media/...
```

Если файл называется кириллицей, для надёжности лучше переименовать его латиницей, например:

```text
avatar.jpg
diagram.png
```

### 5. Frontend показывает старые данные

Можно сделать hard reload:

```text
Cmd + Shift + R
```

Также можно выйти из аккаунта и войти снова.

### 6. Ошибка `HTTP 400` при сохранении квиза

Обычно это ошибка валидации данных. Нужно проверить:

- у каждого вопроса есть текст;
- у вопроса есть правильный ответ;
- у одиночного выбора ровно один правильный вариант;
- у множественного выбора хотя бы один правильный вариант;
- у краткого ответа заполнен правильный текст;
- у числового ответа заполнено правильное число;
- `media_url` является корректной ссылкой или относительным путём `/media/...`.

## Ограничения текущей версии

В текущей версии реализован основной рабочий сценарий самостоятельного прохождения. Некоторые функции заложены архитектурно, но не реализованы полностью.

### Live-режим

В модели и интерфейсе есть режим `Live-прохождение`, однако полноценные live-комнаты пока не реализованы.

Для полноценного live-режима нужны:

- live-сессии;
- лобби участников;
- старт преподавателем;
- синхронизация текущего вопроса;
- live-табло;
- WebSocket или polling.

### Расширенные типы вопросов

Реализованы основные типы:

- одиночный выбор;
- множественный выбор;
- верно/неверно;
- краткий ответ;
- числовой ответ.

Потенциально можно добавить:

- соответствие;
- упорядочивание;
- развёрнутый ответ;
- загрузку файла как ответ;
- формульный ответ.

### Production-настройки

Текущий проект настроен для локального запуска и демонстрации. Для production нужно дополнительно настроить:

- `DEBUG=False`;
- безопасный `SECRET_KEY`;
- корректные `ALLOWED_HOSTS`;
- HTTPS;
- SMTP для отправки писем;
- rate limiting;
- хранение media в объектном хранилище;
- production-сборку frontend;
- production WSGI/ASGI server;
- резервное копирование базы данных.

## Планы развития

Возможные направления дальнейшей разработки:

1. Полноценный live-режим:
   - комнаты;
   - общий старт;
   - синхронизация вопросов;
   - live-табло.

2. Курсы и группы:
   - преподаватель создаёт курс;
   - студенты записываются в курс;
   - аналитика строится по группе.

3. Расширенная аналитика:
   - тепловая карта ошибок;
   - аналитика по темам;
   - аналитика по целям обучения;
   - дискриминативность вопросов.

4. Новые типы вопросов:
   - соответствие;
   - упорядочивание;
   - развёрнутый ответ;
   - файл как ответ.

5. Экспорт:
   - XLSX;
   - PDF-сводки;
   - расширенный CSV.

6. Безопасность:
   - rate limit;
   - аудит действий;
   - ограничение типов загружаемых файлов;
   - защита от подозрительной активности.

7. Интеграции:
   - email SMTP;
   - SSO;
   - импорт из внешних LMS;
   - QTI-подобный обмен вопросами.

## Краткое описание для защиты

QuizCraft — это веб-платформа для создания и проведения учебных квизов. В системе реализованы регистрация, вход, восстановление пароля, профиль пользователя, банк вопросов, разные типы вопросов, создание публичных и приватных квизов, таймеры, лимиты попыток, рандомизация, разные политики обратной связи, прохождение попыток, автоматическая проверка, статистика пользователя и аналитика результатов по квизу. Проект запускается через Docker Compose и включает backend на Django REST Framework, frontend на React и базу данных PostgreSQL.
