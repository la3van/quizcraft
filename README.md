# QuizCraft

QuizCraft - веб-платформа для создания, прохождения и анализа квизов и викторин.

Проект ориентирован на два близких, но разных сценария:

- **Квизы** - учебные и проверочные материалы: например, «Математика 5 класс», «Основы Python», «Тест по HTML».
- **Викторины** - развлекательные подборки вопросов: например, кино, музыка, спорт, общая эрудиция.

Система позволяет создавать материалы, наполнять их вопросами из общего банка, проходить тесты по ссылке или коду доступа, продолжать активные попытки, автоматически завершать попытки по таймеру и смотреть аналитику результатов.

---

## Основные возможности

### 1. Квизы и викторины

В проекте есть явное разделение материалов по типу:

- `quiz` - учебный квиз;
- `trivia` - развлекательная викторина.

Для пользователя это разделение видно в интерфейсе:

- отдельный каталог учебных квизов;
- отдельный каталог развлекательных викторин;
- отдельные страницы создания квиза и викторины;
- отдельная история попыток по квизам и по викторинам;
- бейдж на карточке показывает тип материала.

При этом технически квизы и викторины используют общую модель данных. Это уменьшает дублирование кода: у них одинаковая логика вопросов, попыток, результатов, аналитики и QR-кодов, но разный пользовательский смысл.

---

### 2. Создание и редактирование материалов

Редактор позволяет настроить:

- название;
- описание;
- тип материала: квиз или викторина;
- статус публикации: черновик, опубликован, архив;
- видимость: публичный или приватный;
- сложность: лёгкий, средний, сложный;
- режим прохождения: самостоятельный или live;
- ограничение по времени;
- максимальное количество попыток;
- перемешивание вопросов;
- перемешивание вариантов ответа;
- политику показа обратной связи.

Вопросы можно создавать прямо внутри редактора. Для вопроса поддерживаются:

- текст вопроса;
- объяснение после ответа;
- теги;
- цель обучения;
- баллы;
- тип вопроса;
- варианты ответа;
- правильные ответы;
- медиа: файл, аудио, видео или ссылка.

---

### 3. Общий банк вопросов

Банк вопросов сделан общим для всех пользователей.

Это означает:

- все пользователи видят общий список вопросов;
- вопросы можно использовать при создании квизов и викторин;
- редактировать и удалять можно только свои вопросы;
- вопросы можно импортировать из CSV;
- общий банк можно экспортировать в CSV.

Банк вопросов поддерживает поиск и фильтрацию.

Поиск работает по:

- тексту вопроса;
- объяснению;
- тегам;
- цели обучения;
- теме.

Фильтрация работает по:

- теме;
- типу вопроса;
- типу медиа;
- автору.

---

### 4. Поиск и фильтрация квизов и викторин

В каталогах квизов и викторин есть поиск по названию и описанию.

Поддерживается обычный и неточный поиск. Например, пользователь может ввести запрос с небольшой ошибкой, и система всё равно попробует найти подходящие материалы.

Также доступны фильтры:

- статус публикации;
- сложность;
- видимость;
- количество элементов на странице: 5, 10 или 20.

Архитектурно поиск устроен так:

1. Frontend собирает значения из формы фильтрации.
2. Frontend отправляет их на backend как query-параметры.
3. Backend фильтрует queryset через Django ORM.
4. Для неточного поиска используется дополнительная проверка похожести текста.

Пример query-параметров:

```text
/api/quizzes/?kind=quiz&search=математика&publish_status=published&difficulty=easy&page_size=10
```

### 5. QR-код и вход по коду

Для каждого квиза или викторины можно открыть окно «QR-код».

В этом окне отображаются:

- QR-код;
- код доступа;
- ссылка для входа;
- кнопка копирования ссылки.

QR-код ведёт на страницу входа по коду:

```text
/join/<access_code>
```

Например:

```text
http://localhost:5173/join/ABC123
```

После открытия такой ссылки код автоматически подставляется на странице входа. Если код относится к викторине, пользователь будет перенаправлен на страницу викторины. Если код относится к квизу, пользователь будет перенаправлен на страницу квиза.

QR-код генерируется на frontend без внешних API и без дополнительных npm-зависимостей.

### 6. Попытки прохождения

Когда пользователь начинает квиз или викторину, создаётся попытка.

У попытки есть:

- уникальный внутренний `id`;
- отображаемый ID вида `ATT-0001`;
- пользователь;
- квиз или викторина;
- время начала;
- дедлайн, если установлен таймер;
- время завершения;
- итоговый балл;
- статус отправки;
- порядок вопросов и вариантов ответа.

Отображаемый ID формируется на frontend из обычного числового `Attempt.id`:

```text
id = 1  -> ATT-0001
id = 12 -> ATT-0012
```

Это не отдельное поле в базе, а удобная визуальная метка.

---

### 7. Продолжение активной попытки

Если пользователь начал прохождение, но не завершил его, попытка остаётся активной.

В списке попыток:

- пока время не истекло, показывается кнопка **«Продолжить»**;
- если попытка завершена, показывается кнопка **«Результат»**.

В каталоге квизов и викторин также учитывается активная попытка:

- если активной попытки нет, кнопка называется **«Начать»**;
- если активная попытка есть и её таймер не истёк, кнопка называется **«Продолжить»**;
- рядом показывается ID активной попытки, например `ATT-0016`.

Ссылка на продолжение выглядит так:

```text
/quizzes/1?attempt=16
```

или для викторины:

```text
/trivia/1?attempt=16
```

---

### 8. Автоматическое завершение по таймеру

Если у квиза или викторины задано ограничение по времени, у попытки появляется `deadline_at`.

Если пользователь находится внутри прохождения и таймер заканчивается, frontend автоматически отправляет попытку.

Если пользователь вышел со страницы, отдельный фоновый воркер не используется. Поэтому применяется ленивое автозавершение:

- при следующем обращении к backend система проверяет просроченные активные попытки;
- если `deadline_at` уже прошёл, попытка автоматически завершается;
- в списке попыток она отображается как **«Завершена»**;
- кнопка меняется с **«Продолжить»** на **«Результат»**.

Страница попыток дополнительно обновляется по ближайшему дедлайну, чтобы интерфейс сам поменял состояние попытки после истечения времени.

---

### 9. История попыток

Есть отдельные страницы:

- мои попытки по квизам;
- мои попытки по викторинам.

На странице попыток отображаются:

- ID попытки;
- тип материала;
- название квиза или викторины;
- результат;
- статус;
- дата;
- действие.

Доступна фильтрация:

- поиск по названию квиза или викторины;
- статус попытки: любая, завершена, в процессе;
- дата от;
- дата до;
- минимальный результат в процентах;
- максимальный результат в процентах.

Для результата используются два ползунка: нижняя и верхняя граница процента.

---

### 10. Результат попытки

После завершения попытки пользователь видит страницу результата.

На ней отображаются:

- итоговый процент;
- набранные баллы;
- количество правильных ответов;
- дата завершения;
- подробный разбор ответов.

В разборе видно:

- текст вопроса;
- ответ пользователя;
- правильный ответ;
- начисленные баллы;
- объяснение после ответа, если оно задано.

---

### 11. Аналитика квиза или викторины

Для автора материала доступна аналитика.

Страница аналитики показывает:

- количество завершённых попыток;
- средний результат;
- средний балл;
- минимальный и максимальный процент;
- сложные вопросы;
- попытки участников.

В списке попыток участников можно искать и фильтровать данные:

- по пользователю;
- по проценту результата;
- по дате.

Также доступен экспорт аналитики в CSV.

## Основные модели данных

### Quiz

Хранит квиз или викторину.

Ключевые поля:

- `title` - название;
- `description` - описание;
- `kind` - тип: квиз или викторина;
- `author` - автор;
- `visibility` - публичный или приватный;
- `difficulty` - сложность;
- `publish_status` - статус публикации;
- `access_code` - код доступа;
- `time_limit_minutes` - лимит времени;
- `max_attempts` - максимальное количество попыток;
- `shuffle_questions` - перемешивание вопросов;
- `shuffle_options` - перемешивание вариантов;
- `feedback_policy` - политика обратной связи;
- `delivery_mode` - самостоятельный или live-режим.

### Question

Хранит вопрос.

Вопрос может быть:

- частью конкретного квиза;
- элементом общего банка вопросов.

Ключевые поля:

- `text` - текст вопроса;
- `explanation` - объяснение;
- `tags` - теги;
- `learning_goal` - учебная цель;
- `type` - тип вопроса;
- `points` - баллы;
- `media_kind` - тип медиа;
- `media_file` - загруженный файл;
- `media_url` - ссылка на медиа.

### Option

Хранит вариант ответа для вопроса с выбором.

### Attempt

Хранит попытку прохождения.

Ключевые поля:

- `quiz` - квиз или викторина;
- `user` - пользователь;
- `started_at` - начало;
- `deadline_at` - дедлайн;
- `finished_at` - завершение;
- `score` - баллы;
- `is_submitted` - отправлена ли попытка;
- `question_order` - порядок вопросов;
- `option_order` - порядок вариантов.

### Answer

Хранит ответ пользователя на конкретный вопрос в конкретной попытке.

## Основные пользовательские сценарии

### Сценарий 1. Создание учебного квиза

1. Пользователь открывает раздел **Квизы**.
2. Нажимает **Создать квиз**.
3. Заполняет название и описание.
4. Выбирает статус, сложность, видимость и режим прохождения.
5. Добавляет вопросы вручную или использует общий банк вопросов.
6. Сохраняет квиз.
7. Публикует квиз и делится кодом или QR-кодом.

### Сценарий 2. Создание развлекательной викторины

1. Пользователь открывает раздел **Викторины**.
2. Нажимает **Создать викторину**.
3. Добавляет вопросы развлекательного характера.
4. Настраивает таймер, попытки и публикацию.
5. Делится ссылкой или QR-кодом.

### Сценарий 3. Прохождение по QR-коду

1. Участник сканирует QR-код.
2. Открывается страница входа по коду.
3. Код уже подставлен автоматически.
4. Участник переходит к квизу или викторине.
5. Начинает попытку.
6. После завершения видит результат.

### Сценарий 4. Продолжение попытки

1. Участник начинает попытку.
2. Возвращается в список попыток.
3. Если время не истекло, видит кнопку **Продолжить**.
4. Нажимает её и возвращается к той же попытке.
5. Если время истекло, попытка автоматически завершается, и доступна кнопка **Результат**.

### Сценарий 5. Просмотр аналитики

1. Автор открывает карточку своего квиза или викторины.
2. Нажимает **Аналитика**.
3. Видит общую статистику, сложные вопросы и попытки участников.
4. При необходимости фильтрует попытки по пользователю, дате и результату.
5. Экспортирует данные в CSV.

## Особенности реализации

- Квизы и викторины используют общую backend-модель, но различаются полем `kind`.
- Поиск и фильтрация выполняются на backend по query-параметрам, а frontend только управляет формой фильтров.
- QR-код строится на frontend и не требует стороннего сервиса.
- Активная попытка не создаётся повторно: если у пользователя уже есть незавершённая попытка и её время не истекло, система возвращает её.
- Просроченные попытки завершаются автоматически при следующем обращении к backend.
- Общий банк вопросов виден всем пользователям, но редактирование и удаление ограничено автором вопроса.

- фоновые задачи для мгновенного завершения просроченных попыток;
- уведомления;
- комментарии или обсуждения вопросов;
- расширенную аналитику по темам и учебным целям;
- экспорт результатов в PDF.

На текущем этапе эти функции не обязательны, потому что уже реализованы основные сценарии платформы: создание, публикация, прохождение, продолжение попыток, QR-доступ, поиск, фильтрация, общий банк вопросов и аналитика.
